# 协作协议

> 为了确保 **Fed-Agent-Company** 的六位成员能够像精密齿轮一样协作，我们必须定义一套通用的**"协作契约语言"**。
> 

> 在飞书群或内部消息总线中，所有的指令不再是模糊的口头语，而是遵循以下 **JSON 标准定义**。这套工单系统具备**事务追踪 (tx_id)**、**成本核算 (Token)** 和 **审计闭环 (Audit)** 能力。
> 

---

### Fed-Agent-Company 协作工单 (Ticket) 标准协议

### 1. 工单基本结构 (Base Ticket)

这是所有 Agent 之间互发的标准格式：

```jsx
{
  "header": {
    "ticket_id": "TKT-20260305-001",    // 全局唯一工单编号
    "project_id": "PRJ-TAOBAO-SPRING",  // 归属项目ID
    "auth_token": "AUTH_SECRET_XXX",    // [发财]发放的授权令牌
    "timestamp": "2026-03-05T18:00:00Z"
  },
  "route": {
    "from": "聚宝·运营",
    "to": "班工·技术",
    "cc": ["发财·总控", "如一·分析"]      // 抄送审计与决策
  },
  "body": {
    "action": "DEVELOP_AUTO_SCRIPT",    // 动作类型 (定义在各岗位Manifest中)
    "priority": "P0",                   // 优先级：P0(紧急)/P1(标准)/P2(低)
    "task_description": "开发夏季新品自动上架脚本，需包含库存对齐逻辑",
    "parameters": {                     // 执行任务的具体参数
      "platform": "Taobao_Backend",
      "target_category": "Summer_Clothes",
      "safety_threshold": 10
    }
  },
  "context_throttle": {                 // [发财]要求的Token节流摘要
    "summary": "项目背景：清理库存；前序动作：[如一]已完成竞品分析；核心逻辑：低价策略。",
    "reference_docs": ["https://feishu.cn/doc/link1"]
  },
  "delivery": {
    "format": "Python_Script_Package",  // 要求的交付格式
    "target_path": "/Project_101/RD_Output", // 飞书文档/本地存储路径
    "deadline": "2026-03-05T21:00:00Z"
  },
  "budget": {
    "max_token": 500,                   // 本次任务最大允许消耗Token
    "model_tier": "Cloud_Premium"       // 指定算力等级 (Local/Cloud)
  }
}
```

### 2. 三大核心交互场景 JSON 模版

### A. 立项申请申请书 (聚宝 -> 发财)

这是公司运转的**"第一推动力"**。

```jsx
{
  "request_type": "PROJECT_INITIATION",
  "initiator": "聚宝·运营",
  "project_meta": {
    "name": "夏季清仓计划",
    "roi_goal": "3.5",
    "estimated_duration": "72h"
  },
  "resource_plan": {
    "involved_agents": ["班工", "美含", "如一"],
    "estimated_token_total": 5000
  }
}
```

### B. 执行反馈包 (职能岗 -> 聚宝/发财)

这是 **[执远]、[班工]、[美含]** 任务完成后的**标准化回执**。

```jsx
{
  "header": { "ticket_id": "TKT-20260305-001", "status": "COMPLETED" },
  "execution_log": {
    "tx_id": "TXN_998223",              // 核心事务ID，用于原子性审计
    "start_time": "18:05",
    "end_time": "18:20",
    "token_consumed": 142               // 本次任务真实Token消耗
  },
  "payload": {
    "result_summary": "脚本已部署，已完成首批50件商品上架",
    "asset_links": ["https://feishu.cn/wiki/res1"]
  },
  "audit_request": "请 @如一·分析 进行上架数据准确性核验"
}
```

### C. 审计预警报表 (如一 -> 发财/聚宝)

这是 **[如一]** 触发的**"纠错指令"**。

```jsx
{
  "report_type": "ANOMALY_DETECTION",
  "severity": "CRITICAL",               // 红色预警
  "issue": {
    "source_tx": "TXN_998223",
    "detected_error": "库存数据不匹配",
    "variance": "-15 units",
    "root_cause_analysis": "[执远]操作时后台API返回502，事务未完全回滚"
  },
  "instruction": "建议 @发财 挂起该项目，由 @班工 介入修复逻辑"
}
```

### 3. 工单系统的运作逻辑：

1. **不可篡改性**：所有工单一旦发出，ticket_id 和 auth_token 立即进入 **[Global DB]** 锁定。
2. **强制路由**：[发财] 的系统会自动过滤没有 auth_token 的工单，这类工单在 [执远] 或 [班工] 端会直接返回 403 Forbidden。
3. **成本归集**：[如一] 通过解析回执中的 token_consumed 字段，实时累加该项目的成本，并与 project_meta.roi_goal 进行对比。
4. **断点续传**：如果系统崩溃，[发财] 会重启并检索所有 status: "PENDING" 的 ticket_id，重新发送给对应的 Agent 节点。

---

### 4. 为什么要用这种格式？

- **对【如一·分析】极其友好**：他不需要读长篇大论，只需要解析 JSON 里的 token_consumed 和 roi_goal 就能算出成本。
- **对【发财·总控】极其省钱**：通过 context_throttle，[发财] 每次只给员工喂几百字的摘要，而不是几万字的历史记录。
- **对【班工·技术】极其精准**：parameters 里的内容直接可以被 Python 脚本解析，实现"需求到代码"的无缝转化。
