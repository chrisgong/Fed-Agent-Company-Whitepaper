# 经济模型

这份**《Fed-Agent-Company Token 经济与成本治理模型》**是公司长期盈利的生命线。它不再仅仅是一段文字，而是一套完整的**成本核算与资源调度系统**。

其核心目标是：**将 API 支出视为"原材料"，将 Agent 产出视为"商品"，确保每一分 Token 的消耗都能转化为可衡量的业务增量。**

---

# Fed-Agent-Company Token 经济与成本治理模型

## 1. 算力成本分层策略 (Inference Tiering)

我们根据任务的**逻辑复杂度**与**数据敏感度**，将算力分为三个等级，由【发财·总控】动态调度：

### L1：本地零成本层 (Local/Zero-Cost)

- **适用岗位**：【发财】(CEO)、【执远】(执行)、【如一】(数据)。
- **模型**：Ollama 托管的 DeepSeek/Qwen 系列。
- **经济意义**：处理高频操作、事务日志、敏感审计。这部分任务占比公司总任务数的 70%，通过本地运行，**边际成本为零**。

### L2：云端智力层 (Cloud/Standard)

- **适用岗位**：【聚宝】(运营) 的日常决策、【班工】(技术) 的标准代码。
- **模型**：Claude 3.5 Sonnet / GPT-4o-mini。
- **经济意义**：利用云端模型的高逻辑性处理关键节点，确保运营方向不偏离。

### L3：高阶创意层 (Cloud/Premium)

- **适用岗位**：【美含】(内容) 的深度视觉文案、【班工】的复杂架构设计。
- **模型**：GPT-4o / Claude 3.5 Opus。
- **经济意义**：仅用于"非标"的高价值产出，严格限制调用频次。

## 2. "Token 节流阀"核心技术规范 (Throttle Protocol)

【发财·总控】在分派工单时，必须执行以下技术干预以压缩成本：

### 2.1 动态项目摘要 (Dynamic Project Snapshot)

- **逻辑**：严禁 Agent 读取全量历史。由【发财】根据当前 `ticket_id`，从向量库提取不超过 **500 Tokens** 的任务相关快照注入 Context。
- **效果**：单次调用 Token 消耗降低 60%-80%。

### 2.2 工具 Manifest 最小化加载

- **逻辑**：不加载岗位全量 ToolSet。若当前任务仅需"读取数据库"，则不加载"网络搜索"等工具描述。
- **效果**：系统提示词（System Prompt）长度大幅缩减。

### 2.3 强制结构化响应 (JSON Force)

- **逻辑**：强制所有云端 Agent 采用 JSON 响应。
- **效果**：消除 AI "废话"及寒暄产生的 Token 损耗。

---

## 3. 动态预算配额制 (Dynamic Quota Management)

【发财·总控】通过 **[Global DB]** 实时监控各岗位的"钱包"余额：

### 3.1 项目级预算 (Project-Based Quotas)

- 立项时，【聚宝】必须预估该项目的 Token 消耗量。
- 若消耗达到预警线（如 80%），【发财】自动暂停该项目协作，要求【聚宝】提交《成本超支分析》，经用户确认后方可追加。

### 3.2 岗位绩效调节 (Performance-Linked Quota)

- **S 级 Agent (高效/精准)**：【发财】会自动奖励更多 Cloud 算力配额。
- **C 级 Agent (幻觉多/重做率高)**：【发财】会降级其算力，强制其使用本地 L1 级模型进行"自我反思与逻辑自校正"。

---

## 4. 【如一】的成本审计职能 (CFO Audit)

【如一·分析】作为首席财务官，每日 23:00 必须输出《Token 投资报表》：

1. **成本归集**：汇总当日所有 `tx_id` 产生的真实 Token 费用。
2. **异常诊断**：
    - **死循环检测**：监控短时间内异常高频的 API 请求。
    - **冗余分析**：检测 Agent 之间的回复是否包含大量重复信息。
3. **ROI 核算**：计算公式：`Token ROI = (今日产生业务增量 / 今日 API 支出)`。
    - 若 ROI < 1.0（即 API 费超过收益），【如一】有权行使**一票否决权**，要求【聚宝】重新设计业务链路。

---

## 5. 灾备与熔断机制 (Safeguard)

### 5.1 物理熔断

- 在 OpenRouter/OneAPI 后台设置**日消费硬上限**（例如每日 $5）。
- 一旦熔断，【发财】自动将全员切换至"本地低功耗模式"，仅执行基础的发货与库存检查，停止一切云端创作。

### 5.2 缓存复用

- 对于重复性极高的请求（如：【美含】针对同类产品的 SEO 标题优化），系统优先调用本地 Redis 中的语义缓存，避免重复向云端付费。

## 6. 经济模型目标汇总

| 指标 | 目标设定 |
| --- | --- |
| **本地处理率** | > 70% 的 Task 由本地模型完成 |
| **单任务 Token 中位值** | 控制在 1500 Tokens 以下 |
| **数据资产化率** | 每一分钱 Token 必须换回至少一个持久化的 Markdown 资产 |
| **毛利率预警** | 当 Token 成本占毛利比例 > 5% 时触发报警 |
