# 协作与运行

多 Agent 之间通过**标准化工单（Ticket）**与**每日闭环复盘**协同，保证事务可追踪、成本可核算、审计可闭环。本章定义协作协议与运行节奏。

**与 OpenClaw 的边界** ：工单协议（Ticket、tx_id）为**业务层定义**，由 PostgreSQL 与自建服务维护；OpenClaw 负责把来自某 channel 的会话路由到对应 Agent 并回传，**不替代**工单存储与审计。`Cursor IDE + MCP` 作为老板研发专线接入时，仅扩展指令入口与工具调用方式，不改写工单状态机、审计链路与权限边界。

## 二、飞书交互与意图路由

为兼顾**集中治理**与**并发效率**，飞书侧采用「审批流」与「直达流」并存的意图路由机制：

| 场景 | 触发条件 | 路由规则 | 发财角色 |
|------|----------|----------|----------|
| **日常闲聊 / 非项目需求** | 点对点私聊、无标准工单 | 统一先到【发财】判断、澄清与审批 | 前置审批 |
| **跨部门派单** | 需多岗位协作、预算/权限不明 | 统一先到【发财】裁决后再派发 | 前置审批 |
| **项目群内标准工单直达** | 人类或【聚宝】在项目群内**明确 @ 某岗位**，并附带**标准 JSON 工单** | Gateway 直接路由给被 @ 的目标 Agent 执行 | 仅自动 CC，做事后审计 |

### 2.1 直达流生效条件

仅当以下条件**同时满足**时，才允许跳过【发财】的前置拦截：

1. 消息发生在**已立项的项目群**内。
2. 发起者为**人类老板**或**【聚宝】**。
3. 消息中**明确 @ 单一目标岗位**。
4. 附带符合组织资产规范的**标准 JSON 工单**，包含最小必填字段（如 `project_id`、`ticket_type`、`tx_id`/引用、`context_throttle.summary`、`dod`、`cc`）。

### 2.2 直达流执行规则

- Gateway 命中直达条件后，**直接投递**给对应 Agent 执行，不再等待【发财】逐条前置放行。
- 系统必须自动将该工单 **CC 给【发财】**，用于事后审计、预算稽核与异常问责。
- 若工单字段缺失、目标岗位不明确、或涉及高危权限动作，则自动**降级回审批流**，交由【发财】裁决。

## 三、Cursor IDE 通道（老板研发专线）

在现有「点对点私聊 / 项目群 / 飞书文档」之外，补充第四个入口：**Cursor IDE 通道**。

### 3.1 通道定位

- **飞书** ：负责日常商业运营、审批确认、项目群协作与资产归档。
- **Cursor IDE** ：负责底层技术研发、日志检查、脚本调试、工具联调与架构排障，是老板在开发现场的第二指挥中枢。
- **OpenClaw（发财）** ：同时接收来自**飞书 Webhook** 与 **Cursor MCP** 的双端指令，但所有高危动作、标准工单、CC 审计与权限隔离规则保持一致。

### 3.2 运行规则

- 老板在 Cursor 中发起的研发类指令，可通过 MCP 交给 OpenClaw 解析，再路由给【班工】、【如一】或其他目标岗位执行。
- Cursor 通道适合处理「看代码、查日志、做技术评审、联调本地工具、安排测试环境变更」等技术任务，不替代飞书中的经营审批动作。
- 若 Cursor 中的请求涉及生产高危操作、预算变化或跨岗位业务决策，仍必须回到既有审批流，由【发财】执行前置裁决或事后审计。

### 3.3 上下文一致性

- Cursor 通道产生的有效任务，同样必须关联 `project_id`、`tx_id`、执行结果与资产链接，回写 PostgreSQL 与飞书文档。
- 发财需保证飞书与 Cursor 两端看到的是同一条项目上下文，而不是两套割裂会话。

## 四、项目生命周期 (Project Lifecycle)

项目是多 Agent 协作的基本单元。以下为一个项目从发起到归档的完整生命周期，与飞书「点对点 + 项目群」交互逻辑联动：

| 阶段 | 主要动作 | 责任方 | 交互通道 |
|------|----------|--------|----------|
| **1. 需求沟通** | 老板通过**点对点私聊**向发财描述需求 | 人 → 发财 | 点对点 |
| **2. 立项申请** | 发财指派聚宝，聚宝调研后提交《立项申请书》 | 聚宝 → 发财 | 点对点/内部路由 |
| **3. 审批建群** | 发财审批通过，生成 project_id 与 auth_token，创建**项目群**，拉入相关岗位 | 发财 | 项目群（新建） |
| **4. 派单执行** | 聚宝在项目群内发布标准工单；满足直达规则时 Gateway 直接路由给目标 Agent，否则仍由发财裁决后派发；研发类任务可由老板经 Cursor 发起，再进入同一套路由与审计链路 | 聚宝/发财 → 子 Agent | 项目群 / Cursor |
| **5. 过程质控** | 聚宝监控进度，如一审计成本，发财裁决异常 | 聚宝/如一/发财 | 项目群 |
| **6. 交付验收** | 聚宝验收产出，确认 DoD 达标，所有产出写入飞书文档 | 聚宝 | 项目群 + 飞书文档 |
| **7. 结项归档** | 聚宝提交结项复盘，如一出具项目 ROI 报告，所有资产确认入库 | 聚宝/如一 | 项目群 + 飞书文档 |
| **8. 项目关闭** | 发财确认归档完毕，冻结 auth_token，项目群保留存档或解散 | 发财 | 项目群 |

> 日常沟通（非项目类）始终走**点对点**通道；只有创建项目后才建群。项目群内允许「标准 JSON 工单 + 明确 @ 岗位」的直达流，以在不削弱审计的前提下释放并发能力。研发与排障类指令可从 Cursor 发起，但仍并入同一套工单、回写与审计机制。

## 五、每日闭环与复盘 (EOD)

【发财·总控】每日 **23:00** 启动全公司审计，基于 PostgreSQL 与飞书资产做复盘，并发布《经营健康度报表》。

### 5.1 复盘流程概要

1. **数据库事务扫描** ：查询当日 PENDING 事务。
2. **飞书资产核查** ：检查归档是否达标。
3. **调用【如一】审计** ：获取当日 Token 消耗与 ROI。
4. **本地模型研判** ：将上述数据喂给 Qwen-14B 生成复盘总结。
5. **发布与问责** ：推送飞书群；对未完成项点名问责。
6. **协作改进（团队进化）** ：从当日工单流转与返工记录中归纳协作改进项（如某类工单应抄送谁、路由/模板优化建议），写入 memory-lancedb-pro 的 **scope=global** 或更新团队路由/工单约定（见 [03-技术实现](03-技术实现.md) 团队协作进化），供下一日工单与路由使用。

### 5.2 复盘脚本要点 (Python 伪代码)

- 使用 `agent_transactions` 表筛选 `status = 'PENDING' AND created_at::date = 当日`。
- 调用【如一】接口汇总 Token 与 ROI（与 OpenClaw 的会话/API 调用方式由实现决定）。
- 调用飞书 API 发送《经营健康度报表》卡片。
- 对 PENDING 任务向飞书群发送问责提醒。

### 5.3 部署建议

- 确保 Global DB (PostgreSQL) 常驻运行。
- 配置飞书机器人 webhook 与文档 API token。
- Crontab：`00 23 * * * /usr/bin/python3 /path/to/ceo_audit.py`

---

**小结** ：协作协议定义 Agent 间“说什么、怎么传”；每日复盘定义“何时收口、谁负责闭环”，并产出**协作改进项**写入 global 记忆或团队配置，使工单流转与路由随实践持续优化。个体进化与团队协作进化共同保证多 Agent 体系可运维、可审计、可进化。
