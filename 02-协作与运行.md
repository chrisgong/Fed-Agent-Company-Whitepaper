# 协作与运行

多 Agent 之间通过**标准化工单（Ticket）**与**每日闭环复盘**协同，保证事务可追踪、成本可核算、审计可闭环。本章定义协作协议与运行节奏。

**与 OpenClaw 的边界**：工单协议（Ticket、tx_id）为**业务层定义**，由 PostgreSQL 与自建服务维护；OpenClaw 负责把来自某 channel 的会话路由到对应 Agent 并回传，**不替代**工单存储与审计。

---

## 一、协作工单协议 (Ticket)

在飞书群或内部消息总线中，所有指令遵循 **JSON 工单**，具备事务追踪 (tx_id)、成本核算 (Token)、审计闭环 (Audit) 能力。

### 1.1 工单基本结构 (Base Ticket)

```jsx
{
  "header": {
    "ticket_id": "TKT-20260305-001",
    "project_id": "PRJ-TAOBAO-SPRING",
    "auth_token": "AUTH_SECRET_XXX",
    "timestamp": "2026-03-05T18:00:00Z"
  },
  "route": {
    "from": "聚宝·运营",
    "to": "班工·技术",
    "cc": ["发财·总控", "如一·分析"]
  },
  "body": {
    "action": "DEVELOP_AUTO_SCRIPT",
    "priority": "P0",
    "task_description": "开发夏季新品自动上架脚本，需包含库存对齐逻辑",
    "parameters": { "platform": "Taobao_Backend", "target_category": "Summer_Clothes", "safety_threshold": 10 }
  },
  "context_throttle": {
    "summary": "项目背景：清理库存；前序动作：[如一]已完成竞品分析；核心逻辑：低价策略。",
    "reference_docs": ["https://feishu.cn/doc/link1"]
  },
  "delivery": {
    "format": "Python_Script_Package",
    "target_path": "/Project_101/RD_Output",
    "deadline": "2026-03-05T21:00:00Z"
  },
  "budget": { "max_token": 500, "model_tier": "Cloud_Premium" }
}
```

若启用 OpenClaw 的 [memory-lancedb-pro](https://github.com/win4r/memory-lancedb-pro)，`context_throttle.summary` 可由各 Agent 通过 `memory_recall` 按 project_id/scope 检索后生成，实现动态摘要（见 [03-技术实现](03-技术实现.md)）。

### 1.2 三大核心场景

**A. 立项申请 (聚宝 → 发财)**

```jsx
{
  "request_type": "PROJECT_INITIATION",
  "initiator": "聚宝·运营",
  "project_meta": { "name": "夏季清仓计划", "roi_goal": "3.5", "estimated_duration": "72h" },
  "resource_plan": { "involved_agents": ["班工", "美含", "如一"], "estimated_token_total": 5000 }
}
```

**B. 执行反馈 (职能岗 → 聚宝/发财)**

```jsx
{
  "header": { "ticket_id": "TKT-20260305-001", "status": "COMPLETED" },
  "execution_log": { "tx_id": "TXN_998223", "start_time": "18:05", "end_time": "18:20", "token_consumed": 142 },
  "payload": { "result_summary": "脚本已部署，已完成首批50件商品上架", "asset_links": ["https://feishu.cn/wiki/res1"] },
  "audit_request": "请 @如一·分析 进行上架数据准确性核验"
}
```

**C. 审计预警 (如一 → 发财/聚宝)**

```jsx
{
  "report_type": "ANOMALY_DETECTION",
  "severity": "CRITICAL",
  "issue": {
    "source_tx": "TXN_998223",
    "detected_error": "库存数据不匹配",
    "variance": "-15 units",
    "root_cause_analysis": "[执远]操作时后台API返回502，事务未完全回滚"
  },
  "instruction": "建议 @发财 挂起该项目，由 @班工 介入修复逻辑"
}
```

### 1.3 运作逻辑

1. **不可篡改**：工单发出后 ticket_id、auth_token 进入 [Global DB] 锁定。
2. **强制路由**：无 auth_token 的工单在执远/班工端返回 403。
3. **成本归集**：【如一】解析 token_consumed，与 project_meta.roi_goal 对比。
4. **断点续传**：崩溃后【发财】检索 status: "PENDING" 的 ticket_id 重新下发。

## 二、每日闭环与复盘 (EOD)

【发财·总控】每日 **23:00** 启动全公司审计，基于 PostgreSQL 与飞书资产做复盘，并发布《经营健康度报表》。

### 2.1 复盘流程概要

1. **数据库事务扫描**：查询当日 PENDING 事务。
2. **飞书资产核查**：检查归档是否达标。
3. **调用【如一】审计**：获取当日 Token 消耗与 ROI。
4. **本地模型研判**：将上述数据喂给 Qwen-14B 生成复盘总结。
5. **发布与问责**：推送飞书群；对未完成项点名问责。
6. **协作改进（团队进化）**：从当日工单流转与返工记录中归纳协作改进项（如某类工单应抄送谁、路由/模板优化建议），写入 memory-lancedb-pro 的 **scope=global** 或更新团队路由/工单约定（见 [03-技术实现](03-技术实现.md) 团队协作进化），供下一日工单与路由使用。

### 2.2 复盘脚本要点 (Python 伪代码)

- 使用 `agent_transactions` 表筛选 `status = 'PENDING' AND created_at::date = 当日`。
- 调用【如一】接口汇总 Token 与 ROI（与 OpenClaw 的会话/API 调用方式由实现决定）。
- 调用飞书 API 发送《经营健康度报表》卡片。
- 对 PENDING 任务向飞书群发送问责提醒。

### 2.3 部署建议

- 确保 Global DB (PostgreSQL) 常驻运行。
- 配置飞书机器人 webhook 与文档 API token。
- Crontab：`00 23 * * * /usr/bin/python3 /path/to/ceo_audit.py`

---

**小结**：协作协议定义 Agent 间“说什么、怎么传”；每日复盘定义“何时收口、谁负责闭环”，并产出**协作改进项**写入 global 记忆或团队配置，使工单流转与路由随实践持续优化。个体进化与团队协作进化共同保证多 Agent 体系可运维、可审计、可进化。
