# 05 组织资产：工单与协作规范

> **版本**: v1.0 | **最后更新**: 2026-03-06

本文档为 Fed-Agent-Company 的**组织级资产**，定义协作工单协议、内部协作风格、工单合规规则与汇报模板，供所有 Agent 与实现方统一遵循。修订后需同步至 [02-协作与运行](02-协作与运行.md) 的引用说明。

## 一、协作工单协议 (Ticket)

在飞书项目群或内部消息总线中，所有指令遵循 **JSON 工单**，具备事务追踪 (tx_id)、成本核算 (Token)、审计闭环 (Audit) 能力。

### 1.1 工单基本结构 (Base Ticket)

```json
{
  "header": {
    "ticket_id": "TKT-20260305-001",
    "project_id": "PRJ-TAOBAO-SPRING",
    "auth_token": "AUTH_SECRET_XXX",
    "timestamp": "2026-03-05T18:00:00Z"
  },
  "route": {
    "from": "聚宝·运营",
    "to": "班工·技术",
    "cc": ["发财·总控", "如一·分析"]
  },
  "body": {
    "action": "DEVELOP_AUTO_SCRIPT",
    "priority": "P0",
    "task_description": "开发夏季新品自动上架脚本，需包含库存对齐逻辑",
    "parameters": { "platform": "Taobao_Backend", "target_category": "Summer_Clothes", "safety_threshold": 10 }
  },
  "context_throttle": {
    "summary": "项目背景：清理库存；前序动作：[如一]已完成竞品分析；核心逻辑：低价策略。",
    "reference_docs": ["https://feishu.cn/doc/link1"]
  },
  "delivery": {
    "format": "Python_Script_Package",
    "target_path": "/Project_101/RD_Output",
    "deadline": "2026-03-05T21:00:00Z"
  },
  "budget": { "max_token": 500, "model_tier": "Cloud_Premium" }
}
```

若启用 [memory-lancedb-pro](https://github.com/win4r/memory-lancedb-pro)，`context_throttle.summary` 可由各 Agent 通过 `memory_recall` 按 project_id/scope 检索后生成（见 [03-技术实现](03-技术实现.md)）。

### 1.2 三大核心场景

**A. 立项申请 (聚宝 → 发财)**

```json
{
  "request_type": "PROJECT_INITIATION",
  "initiator": "聚宝·运营",
  "project_meta": { "name": "夏季清仓计划", "roi_goal": "3.5", "estimated_duration": "72h" },
  "resource_plan": { "involved_agents": ["班工", "美含", "如一"], "estimated_token_total": 5000 }
}
```

**B. 执行反馈 (职能岗 → 聚宝/发财)**

```json
{
  "header": { "ticket_id": "TKT-20260305-001", "status": "COMPLETED" },
  "execution_log": { "tx_id": "TXN_998223", "start_time": "18:05", "end_time": "18:20", "token_consumed": 142 },
  "payload": { "result_summary": "脚本已部署，已完成首批50件商品上架", "asset_links": ["https://feishu.cn/wiki/res1"] },
  "audit_request": "请 @如一·分析 进行上架数据准确性核验"
}
```

**C. 审计预警 (如一 → 发财/聚宝)**

```json
{
  "report_type": "ANOMALY_DETECTION",
  "severity": "CRITICAL",
  "issue": {
    "source_tx": "TXN_998223",
    "detected_error": "库存数据不匹配",
    "variance": "-15 units",
    "root_cause_analysis": "[执远]操作时后台API返回502，事务未完全回滚"
  },
  "instruction": "建议 @发财 挂起该项目，由 @班工 介入修复逻辑"
}
```

### 1.3 运作逻辑

1. **不可篡改** ：工单发出后 ticket_id、auth_token 进入 [Global DB] 锁定。
2. **强制路由** ：无 auth_token 的工单在执远/班工端返回 403。
3. **成本归集** ：【如一】解析 token_consumed，与 project_meta.roi_goal 对比。
4. **断点续传** ：崩溃后【发财】检索 status: "PENDING" 的 ticket_id 重新下发。

## 二、内部协作风格

- **用语** ：简洁、客观，避免模糊表述；岗位间称呼使用规范代号（发财·总控、聚宝·运营、执远·执行、班工·技术、美含·内容、如一·分析）。
- **结构** ：工单与汇报均采用「结论/请求先行，再附依据或详情」；长说明使用分条或表格，便于检索与审计。
- **抄送与可见性** ：涉及成本、合规、跨岗交付的工单必须 cc 发财·总控；涉及数据与审计的须 cc 如一·分析；具体规则见 [三、合规规则（工单结构）](#三合规规则工单结构)。
- **响应时效** ：收到 @ 或工单后，在项目群内确认接收（如「已接单 TKT-xxx」）；执行中遇阻塞 2 小时内于群内说明并 @ 派单方。
- **争议与升级** ：岗位间对需求或交付标准有分歧时，在工单或群内标明「待发财裁决」，由发财在点对点或群内给出结论并更新工单。

## 三、合规规则（工单结构）

以下为工单**必须满足**的合规要求，系统或人工校验未通过时不得进入执行队列。

### 3.1 必填字段

| 层级 | 字段 | 说明 |
|------|------|------|
| header | ticket_id | 全局唯一，建议格式 `TKT-YYYYMMDD-序号` |
| header | project_id | 所属项目，与项目群/Global DB 一致 |
| header | auth_token | 有效授权令牌，否则 403 |
| header | timestamp | ISO 8601 |
| route | from, to | 发起方与执行方岗位 |
| body | action | 动作类型（与动作枚举一致） |
| body | task_description | 任务描述，非空 |
| delivery | deadline | 截止时间，ISO 8601 |

### 3.2 抄送 (cc) 规则

- 所有工单 **to** 非发财时，**cc** 必须包含 **发财·总控**。
- 涉及数据统计、ROI、成本、审计结论的工单，**cc** 必须包含 **如一·分析**。
- 涉及跨岗交付（如班工→执远、美含→聚宝）时，**cc** 须包含接收方及发财。

### 3.3 禁止项

- 不得省略 auth_token 或使用过期/无效 token。
- 不得在 task_description 或 parameters 中携带未脱敏的客户隐私、账号密码、支付信息。
- 不得为规避审计而拆分本应为一单的工单（以业务语义为准）。

### 3.4 校验与驳回

- 系统在入队前校验 header/route/body 必填项及 cc 规则；不合规工单返回 400 及错误码，由发起方修正后重发。
- 【如一】在审计中若发现历史工单存在合规瑕疵，在飞书账本「审计建议」中记录，并建议【发财】收紧对应类型的校验规则。

## 四、汇报模板

以下模板用于项目群内**进度汇报、结项汇报、异常汇报**，所有 Agent 输出汇报时优先采用对应模板，便于人类与【发财】快速解析。

### 4.1 进度汇报（进行中）

```text
[进度] TKT-{ticket_id} | 执行方：{岗位} | 状态：{进行中/阻塞/待确认}
- 已完成：{简要列表}
- 进行中：{当前步骤}
- 预计完成：{时间}
- 阻塞/风险：{无或简要说明}
```

### 4.2 结项汇报（完成）

```text
[结项] TKT-{ticket_id} | 执行方：{岗位} | 状态：已完成
- 交付摘要：{1～2 句}
- 产出链接：{飞书文档/多维表链接}
- 消耗：Token {数值}，耗时 {时长}
- 审计请求：{若有，如「请 @如一·分析 核验数据」}
```

### 4.3 异常汇报（问题/预警）

```text
[异常] 严重程度：{P0/P1/P2} | 来源 TKT/tx：{id}
- 现象：{一句话}
- 影响：{范围与后果}
- 建议：{如「挂起项目」「@班工 修复」}
- 根因/备注：{可选}
```

### 4.4 经营/成本汇报（如一·分析 专用）

```text
[经营分析] 关键指标: {XXX} | 今日 Token 成本: ￥{XXX} | 今日 ROI: {XXX} | 趋势: [上升/下降] | 预警: [有/无] | 详细账本: [飞书多维表格链接]
```
