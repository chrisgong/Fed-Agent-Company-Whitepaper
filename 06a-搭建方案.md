# 搭建方案

> 所属：[06 运营方案](06-运营方案.md) > 第一部分

## 一、数据来源全景

运营全链路涉及 **8 大外部数据源/操作平台**，按实现难度分为三个层级。

### 1.1 已有成熟通道（可直接对接）

| 数据源 | 类型 | 用途 | 访问方式 | 负责 Agent |
|--------|------|------|----------|------------|
| **飞书文档/多维表** | SaaS API | 蓝海词协作、数据归档、汇报、工单 | OpenClaw 原生飞书 Channel + 飞书开放 API（文档读写、多维表 CRUD） | 全员 |
| **蓝海词词表** | 文件（Excel/CSV） | 选品起点 | 人工上传至飞书文档 → Agent 通过飞书 API 读取 | 聚宝读取 |
| **[张飞搬家](https://item.jnesoft.com/home)** | 第三方铺货工具 | 1688 商品一键搬家到淘宝店铺 | 提交 1688 商品链接 → 自动同步标题/主图/详情/SKU 到淘宝 | 聚宝/执远 |

飞书是整个体系的协作中枢，OpenClaw 已原生支持飞书 Channel。飞书开放平台提供文档、多维表（Bitable）、消息等完整 API，Agent 可通过 Tool 调用。蓝海词表由人上传到飞书文档，Agent 侧只需「读取」能力，实现门槛低。张飞搬家作为 1688→淘宝的铺货桥梁，省去人工逐条创建商品的环节，搬家后的商品页作为底稿再由美含优化。

### 1.2 需 Playwright 浏览器自动化（执远核心战场）

| 数据源 | 类型 | 用途 | 关键难点 |
|--------|------|------|----------|
| **1688.com** | 电商网页 | 找货选品 | 登录态维护、反爬验证码、页面结构变动 |
| **哈士奇插件** | Chrome 扩展 | 1688 数据增强（复购率等） | 需在 Chromium 中加载扩展，读取插件注入的 DOM 数据 |
| **千牛后台** | 电商网页 | 商品管理（上下架/改价/库存）、营销推广（直通车/万相台调价调词）、优惠券/活动设置 | 登录态（二次验证/滑块）、操作风控、页面改版、推广出价精度到分 |

千牛是淘宝/天猫卖家的统一工作台，商品管理与营销推广（直通车、万相台）都在千牛内操作。这些平台都没有对外开放 API，必须走浏览器自动化，也是执远跑在宿主机（非 Docker）的根本原因。

### 1.3 需数据加工中转（如一的工作）

| 数据源 | 类型 | 真实来源 | 落地路径 |
|--------|------|----------|----------|
| **生意参谋** | 淘宝数据平台 | sycm.taobao.com | 执远 Playwright 抓取 → PostgreSQL → 如一分析 → 飞书多维表 |
| **客服端数据** | 千牛/评价管理 | 淘宝后台 | 执远 Playwright 抓取 → 如一分析 |
| **如一数据看板** | 内部产物 | 上述原始数据加工 | 如一 Agent 分析后写入飞书多维表 |

生意参谋没有对外 API，实际路径为 **执远定时抓取 → 结构化数据存 DB → 如一读取分析 → 产出写入飞书多维表**。

## 三、关键技术要点

### 3.1 登录态管理

千牛（含商品管理、直通车、万相台、生意参谋入口）共享一套登录态，1688 独立一套。需持久化 Cookie/Session，处理二次验证（短信/扫码/滑块）。

**异常处理协议** ：执远在操作任何平台时若遇到登录态失效或触发验证，**立即暂停当前任务**，在对应店铺的飞书项目群内发送通知，等待人工远程登录宿主机处理。人工完成验证后执远自动检测恢复，继续执行暂停的任务队列。详见 [01 执远宪法](01-角色与宪法.md) 中的「登录态管理」协议。

### 3.2 哈士奇插件兼容性

Playwright 的 Chromium 可通过 `--load-extension` 加载 Chrome 扩展。建议**优先用有头模式（headed）运行**，确保哈士奇插件正常加载并渲染数据。

### 3.3 反爬与风控

1688 和淘宝都有反爬机制。执远的 Playwright 脚本须内置随机延迟（2-5 秒/操作）和人类行为模拟（鼠标轨迹、滚动节奏），降低被风控拦截的概率。

**强制执行漏斗模型** ：严禁对单条库存/商品检查频繁开启浏览器。必须采用「**千牛/1688 后台批量导出全店 CSV 报表 → 本地脚本比对差异 → 仅针对差异项开启 UI 操作**」的漏斗模型：

1. **批量导出** ：优先从千牛、生意参谋、1688 后台导出商品、库存、价格、订单等 CSV/Excel 报表。
2. **本地比对** ：由【班工】提供本地脚本进行字段级 diff，比对出新增、缺失、异常与需修复项。
3. **差异执行** ：仅对差异项调用 Playwright/UI 自动化，避免对全店逐条打开页面造成反爬与风控压力。
4. **审计回写** ：差异修复结果回写 PostgreSQL 与飞书文档，保留批次操作快照。

**内存释放防线** ：【班工】开发的所有 RPA / MCP Server 浏览器脚本，必须把 Playwright 会话视为“阅后即焚”的一次性资源：

1. 统一使用严格上下文管理器（如 `with sync_playwright() as p:`）包裹任务级生命周期。
2. 浏览器允许按**任务/批次**复用，不要求每个单条操作后立刻重开；但在任务完成、异常、超时、验证码失败或人工接管前，必须强制执行 `context.close()` 与 `browser.close()`。
3. 严禁让无主 `context`、失控 `browser` 或残留标签页常驻宿主机内存；发现僵尸浏览器标签页视为违规实现。

**验证码处置** ：【班工】必须接入第三方打码平台 API 处理滑块/图形验证；若连续失败或命中高危风控，则立即暂停任务并转人工远程介入。

### 3.4 HITL 介入点

运营手册中明确「批量改价/退款」等高危操作需 HITL。执远在 Playwright 操作链路中执行高危操作前暂停，通过飞书发审批卡片给发财，等待人工确认后再继续。

## 四、落地优先级

| 优先级 | 建设项 | 说明 | 前置依赖 |
|--------|--------|------|----------|
| **P0** | 飞书 API 对接（文档读写 + 多维表） | 所有 Agent 的输出归档通道，必须先打通 | OpenClaw 飞书 Channel |
| **P0** | 1688 Playwright 找货链路 | 核心商业模式的起点，聚宝选完词后立刻要用 | 执远 Playwright 环境 + 哈士奇插件 |
| **P1** | 千牛商品管理操作（上下架、改价、优惠券） | 上新流程的执行环节 | 千牛登录态管理 |
| **P1** | 生意参谋数据抓取 → 如一分析链路 | 日常巡检和诊断的数据基础 | 千牛登录态 + PostgreSQL |
| **P2** | 千牛营销推广操作（直通车/万相台） | 推广调优，前期可人工操作、AI 出建议 | 千牛登录态（共享） |
| **P2** | 客服数据抓取 | 前期可人工速览，后期自动化 | 千牛登录态 |
